## 一、文档定位

                              - 工作流说明
```

**06 是"桥梁"而非"数据源"**：
- 分md（00-05, 07）是**权威数据源**——所有内容最终来自这些文件
- 06 是**骨架映射**——定义每个 uid 条目对应哪个分md的哪部分内容
- 它不重复存储已在分md中的内容，而是**引用**和**指引**
- 对于尚未在分md中详细展开的条目（uid 31-39），06 提供**内容生成指引**
- 对于已在分md中有详细内容的条目（uid 40-42，来自03_变量系统和05_事件系统），06 提供**提取与格式化指南**
- uid 31-42 的内容实际由 build_eldoria.py 中内联定义生成

### 1.2 固定工作流

```
1. 修改分md（00-05, 07）中的内容
2. 如有需要，更新 06_条目规划与格式.md 中的映射关系
3. 运行 build_eldoria.py 重建 Eldoria.json
4. 将 Eldoria.json 导入 SillyTavern
```

**重要规则**：
- **永远不要直接编辑 `Eldoria.json`**——它是一个生成产物，所有手动修改都会在下一次 build 时丢失
- **分md 是唯一的数据源**——所有内容变更必须从分md开始
- **06 是唯一的条目规划文档**——新增/删除/修改条目时，必须同步更新本文档

## 三、单个条目字段规范表

### 3.1 完整字段列表

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `uid` | number | 从0递增 | 条目的唯一标识符，不可重复，不可跳跃 |
| `key` | array of strings | — | 触发关键词数组，混合中英文，小写英文 |
| `content` | string | — | 条目的核心内容文本，使用 `\n` 作为 JSON 换行 |
| `comment` | string | — | 条目的中文描述性说明，格式为 `【类别】简述` |
| `extension` | object | `{}` | 扩展字段，始终为空对象 |
| `constant` | boolean | `false` | 是否始终注入（仅 uid 0 和 uid 6 为 `true`） |
| `selective` | boolean | `false` | 是否选择性触发，始终为 `false` |
| `secondary` | boolean | `true` | 是否为次要条目，始终为 `true` |
| `probability` | number | `100` | 触发概率百分比，核心条目 `100`，路线专属条目 `80` |
| `order` | number | — | 排序优先级，越小越靠前 |
| `position` | string | `"1000"` | 注入位置，始终为 `"1000"`（before_char） |
| `use_regex` | boolean | `false` | 是否对 key 使用正则匹配，始终为 `false` |
| `match_ratio` | number | `0.6` | 关键词匹配比例阈值，始终为 `0.6` |
| `case_sensitive` | boolean | `false` | 是否大小写敏感，始终为 `false` |
| `automation` | boolean | `false` | 是否由自动化触发，始终为 `false` |
| `group` | string | `""` | 条目分组名，始终为空字符串 |
| `group_other` | boolean | `false` | 是否与其他条目分组，始终为 `false` |
| `prevent_recursion` | boolean | `false` | 是否防止递归注入，始终为 `false` |
| `delay_until` | string | `""` | 延迟触发条件，始终为空字符串 |
| `overrides` | object | `{}` | 覆盖配置，始终为空对象 |

### 3.2 关键字段的详细说明

#### uid
- 从 0 开始，严格递增，不可跳跃
- 新增条目时，从当前最大 uid + 1 开始
- 删除条目时，后续 uid 不重新编号（保持稳定）

#### key（关键词数组）
- 类型：`string[]`（字符串数组），不是单个字符串
- 混合中英文，英文使用小写
- 每条 3-8 个关键词
- 关键词应该是对话中用户可能自然使用的词
- 示例：`["seraphina", "塞拉菲娜", "精灵", "守护者", "身份", "天使血脉"]`

#### content（内容文本）
- 使用 `\n` 作为换行符（JSON 转义）
- 使用中文书写
- 结构化"属性: 描述"风格
- 长度取决于条目类型（见第四节）

#### comment（描述性说明）
- 格式：`【类别】具体描述`
- 类别示例：`【世界总览】`、`【角色核心】`、`【地点】`、`【魔法系统】`、`【纯爱路线】`、`【NTRS路线】`、`【被动NTR】`、`【变量系统】`、`【事件系统】`
- 用于人类阅读和理解该条目的作用

#### constant
- 仅 uid 0（世界总览）和 uid 6（世界观总览）为 `true`
- 这两个条目会被始终注入到对话上下文中
- 其他所有条目均为 `false`，由关键词触发

#### probability
- 核心设定条目（角色、地点、魔法、纯爱路线）：`100`
- 路线专属条目（NTRS、被动NTR）：`80`
- 所有路线共存，由玩家选择决定走向

#### order（排序优先级）
按以下规则分配：

| order 范围 | 条目类别 | constant |
|-----------|---------|----------|
| 0 | 世界设定总览 | true |
| 10-20 | Seraphina 基础设定 | false |
| 30-40 | 黎恩设定 | false |
| 50-60 | 其他角色 | false |
| 70-80 | 地点设定 | false |
| 90-100 | 魔法系统 | false |
| 110-150 | 纯爱路线 | false |
| 160-200 | NTRS 路线 | false |
| 210-240 | 被动 NTR 路线 | false |
| 250-270 | 变量系统 | false |
| 280-300 | 事件系统 | false |

## 四、content 字段写作规范

### 4.1 整体风格

- **结构化描述**——不是散文，不是对话
- 使用简单的描述句
- 用冒号分隔：`属性名: 描述内容`
- 多用简短段落，每个段落有明确主题
- 使用 `\n` 作为换行符（JSON 转义）

### 4.2 格式约定

```
【条目标题 — 副标题】

段落标题:
- 要点一: 描述内容
- 要点二: 描述内容
  - 子要点: 进一步描述

1. 编号项:
   - 子项: 描述
   - 子项: 描述

关键属性: 描述内容
```

### 4.3 变量与占位符

- 使用 `{{user}}` 指代玩家（黎恩舒华泽）
- 所有角色名使用中文全名（如"黎恩舒华泽"、"亚莉莎莱恩福尔特"）
- 关键英文名在首次出现时附带（如"Seraphina（塞拉菲娜）"）

### 4.4 路线分支描述

在同一条目中使用清晰的区段区分不同路线：

```
纯爱路线:
- 描述内容
- 描述内容

NTRS路线:
- 描述内容
- 描述内容

被动NTR:
- 描述内容
- 描述内容
```

**不要使用**条件判断语法（如 `<% if %>`）——世界书不支持这种语法。
使用简单的文字说明"如果XX则YY"。

### 4.5 内容长度

| 条目类型 | 推荐长度 | 说明 |
|---------|---------|------|
| 基础设定条目 | 500-1500 字 | 世界观、魔法系统、地点等 |
| 角色条目 | 800-2500 字 | Seraphina、黎恩、NPC 等 |
| 路线条目 | 800-2500 字 | 纯爱/NTRS/被动NTR 的关系阶段 |
| 变量/事件条目 | 500-1500 字 | 变量系统、事件系统 |

### 4.6 具体示例

```
【地点 — 林间空地（The Glade）】

定位: Eldoria森林中唯一永远安全的地方；Seraphina的居所；故事起点

环境描述:
- 一片被金色光芒环绕的圆形空地
- 中央是一棵巨大的古老橡树，树上刻满精灵符文
- 周围环绕着小溪——溪水散发温暖的金色

纯爱路线:
- 描写温暖、安宁、金色光芒——象征两人关系的纯粹

NTRS路线:
- 同样的金色光芒，但会被用来对比「共享」的场景

## 五、key 关键词设计原则

### 5.1 基本规则

1. **混合中英文**：每条 key 同时包含中文和英文关键词
2. **英文小写**：所有英文关键词使用小写
3. **3-8 个关键词**：每条条目 3-8 个关键词，不宜过多或过少
4. **对话中自然出现**：关键词应该是用户在对话中可能自然使用的词
5. **避免单字泛词**：不使用"他"、"她"、"是"等过于通用的单字词
6. **避免冷门词**：不使用用户不可能输入的词

### 5.2 关键词类型

| 类型 | 示例 | 说明 |
|------|------|------|
| 角色名 | `seraphina`, `塞拉菲娜`, `rean`, `黎恩` | 用户会直接称呼的角色名 |
| 地名 | `林间空地`, `glade`, `低语林地` | 用户会提及的地点 |
| 概念词 | `圣光`, `鬼之力`, `腐化`, `纯爱` | 用户会讨论的主题概念 |
| 关系词 | `恋人`, `守护者`, `契约`, `ntrs` | 用户会描述关系时使用的词 |
| 身份词 | `精灵`, `精灵守护者`, `VII班` | 用户会提及的身份标签 |

### 5.3 关键词设计示例

```
uid 1: ["seraphina", "塞拉菲娜", "精灵", "守护者", "身份", "天使血脉"]
uid 6: ["eldoria", "森林", "世界", "地理", "腐化", "心木", "艾尔多利亚"]
uid 23: ["纯爱", "路线", "信任", "告白", "pure love", "恋爱"]
uid 28: ["ntrs", "坦白", "共享", "欲望", "边界", "协商"]
```

### 5.4 注意事项

- 关键词是触发世界书条目注入的机制——用户输入匹配到任意 key 时，对应条目才会被注入
- 过于泛化的关键词会导致条目在不该触发时被注入，浪费上下文
- 过于冷门的关键词会导致条目永不被触发，浪费条目内容
- `match_ratio` 为 0.6 意味着需要匹配 key 数组中 60% 以上的关键词才会触发
