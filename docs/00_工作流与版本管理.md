## 二、版本号系统

### 2.1 版本号格式

```
Eldoria_V{主版本}.{次版本}.{修订号}
```

| 位 | 名称 | 何时递增 | 示例 |
|---|------|---------|------|
| **主版本** | Major | 重大架构变更、路线重设计、核心设定翻版 | V1.0 → V2.0 |
| **次版本** | Minor | 新增角色、新增事件、修改变量系统 | V1.0 → V1.1 |
| **修订号** | Revision | 文本修正、错别字、内容微调 | V1.0 → V1.0.1 |

### 2.2 版本号的使用位置

| 文件 | 位置 | 格式示例 |
|------|------|---------|
| **build_eldoria.py** | 顶部 `VERSION` 常量 | `VERSION = "V1.0"` |
| **Eldoria_V4.6.2.json** | 构建时写入 `_meta.version` | `"version": "Eldoria_V4.6.2"` |
| **备份文件** | `backup/` 目录 | `Eldoria_V1.0_20260616_233045.json` |
| **发布快照** | 项目根目录 | `Eldoria_V1.0.json`（每次构建自动生成） |
| **00_工作流与版本管理.md** | 下方版本历史表 | 手动记录每次变更 |

### 2.3 版本号变更的触发场景

| 操作类型 | 主版本 | 次版本 | 修订号 | 示例 |
|---------|--------|--------|--------|------|
| 修正错别字 / 标点 | | | +1 | V1.0 → V1.0.1 |
| 调整角色描写措辞 | | | +1 | V1.1 → V1.1.1 |
| 新增一条事件 | | +1 | 0 | V1.0 → V1.1 |
| 新增一个 NPC 角色 | | +1 | 0 | V1.2 → V1.3 |
| 修改变量系统的核心逻辑 | | +1 | 0 | V1.1 → V1.2 |
| 重构三线架构 / 新增路线 | +1 | 0 | 0 | V1.5 → V2.0 |
| 移除 JSON 依赖 / 改用 Python 构建 | +1 | 0 | 0 | V2.0 → V3.0 |

## 三、构建脚本输出详解

### 3.1 每次构建会产生的文件

```
世界书/
├── Eldoria_V{VERSION}.json            ← 唯一主输出文件（每次构建写入/覆盖）
└── backup/
    └── Eldoria_V{VERSION}_{timestamp}.json   ← 时间戳副本
```

### 3.2 文件用途区别

| 文件 | 是否可手动编辑 | 用途 | 保留策略 |
|------|--------------|------|---------|
| **Eldoria_V{VERSION}.json** | ❌ 不可 | SillyTavern 加载的世界书文件，文件名直接体现版本号，在 UI 中一目了然 | 每次构建会覆盖同版本号的文件；不同版本号自然共存为多个文件 |
| **backup/Eldoria_V{VERSION}_{timestamp}.json** | ❌ 不可 | 同一版本号多次重新构建时保留的中间过程副本；也保留每一次构建的时间戳记录 | 保留最近 20 个，旧的可手动删除 |

### 3.3 示例：目录结构看起来像这样

```
世界书/
├── Eldoria_V1.0.json          ← V1.0 原始版本
├── Eldoria_V1.1.json          ← V1.1 新增角色后的版本
├── Eldoria_V2.0.json          ← V2.0 架构重构版本
├── Eldoria_V3.0.json          ← V3.0 当前最新版本（SillyTavern 中加载这一个）
└── backup/
    ├── Eldoria_V3.0_20260616_234000.json
    └── ...（其他时间戳副本）
```

### 3.4 如何回滚到旧版本

```bash
# 场景：V3.0 有问题，想回到 V2.0
# 不需要任何特殊操作——直接在 SillyTavern 中选择加载 Eldoria_V2.0.json

# 如果需要删除当前最新版本的文件：
del Eldoria_V3.0.json
```

## 四、内容编辑的工作流细则

### 4.1 新增一个角色的完整流程

```
1. 在 01_角色档案.md 中添加角色档案
   ├── 基本身份（种族、年龄、性格）
   ├── 与黎恩/Seraphina 的关系
   ├── 在 NTRS / 被动NTR 中的角色定位
   └── 关键变量影响

2. 在 06_条目规划与格式.md 中为该角色分配新的 uid
   └── 例如：uid 110 及以后顺延

3. 在 build_eldoria.py 中:
   a. 更新 VERSION 常量（次版本+1，例如 V1.0 → V1.1）
   b. 在合适的函数中添加新的条目定义
      └── 新增 NPC 角色 → get_uid43_46_entries() 或顺延到新增函数
   c. 在 build() 函数中注册新函数调用

4. 更新辅助文档：
   a. 更新01_角色档案.md（添加新角色的外貌/性格对照表）
   b. 更新05_事件系统 事件-章节映射表.md（添加新角色的登场事件时间点）

5. 运行构建:
   python scripts/build_eldoria.py

6. 记录版本历史到本文档
```

### 4.2 新增一个事件的完整流程

```
1. 在 05_事件系统.md 中添加事件定义
   ├── 触发条件（变量阈值）
   ├── 场景描述（按路线区分：纯爱/NTRS/被动NTR）
   └── 变量影响（事件触发后哪些变量会改变）

2. 在 06_条目规划与格式.md 中为该事件分配新的 uid

3. 在 build_eldoria.py 中:
   a. 更新 VERSION 常量
   b. 在 get_uid47_54_entries() 中添加新条目定义
      └── 或顺延到新的函数

4. 运行构建:
   python scripts/build_eldoria.py

5. 记录版本历史到本文档
```

### 4.3 修正文本/措辞的流程

```
1. 在相关分md文件中修正内容

2. 在 build_eldoria.py 中对应的 entry content 做同样修改
   └── 重点：分md和脚本中的内容必须保持一致

3. 更新 VERSION 常量（修订号+1）
   └── V1.1 → V1.1.1

4. 运行构建

5. 记录版本历史（简短说明修改点）
```

### 4.4 辅助文档维护

项目中除了核心分md文档外，还有以下辅助文档需要同步维护：

| 辅助文档 | 维护时机 | 更新要点 |
|---------|---------|---------|
| `01_角色档案.md` | 新增/修改角色设定后 | 同步更新角色的外貌、性格、核心设定对照表 |
| `05_事件系统 事件-章节映射表.md` | 新增事件/调整章节结构后 | 同步更新时间线节点、事件顺序、路线分支 |
| `剧情时间线可视化.html` | 05_事件系统 事件-章节映射表更新后 | 重新生成或手动更新可视化页面数据 |

**维护原则**：
- 核心分md（00-07）是权威数据源，辅助文档从中派生
- 修改核心文档后，必须检查并同步更新相关辅助文档
- 辅助文档的更新应与核心文档的修改在同一次构建会话中完成
- 版本号递增时，辅助文档的变更也计入版本历史

## 七、常见问题与解决方案

### Q1: 我修改了分md内容，但 Eldoria_V4.7.0.json 没有变化？
**A**: 分md文件是权威数据源的「文档」，但 JSON 的实际构建内容内嵌在 `build_eldoria.py` 的各个 `get_uidXX_YY_entries()` 函数中。你需要：
1. 修改分md（作为权威文档）
2. 在 `build_eldoria.py` 的对应函数中做同样的修改
3. 运行 `python build_eldoria.py`

### Q2: 如何确认我使用的是哪个版本？
**A**: 检查 Eldoria_V4.7.0.json 中的 `_meta.version` 字段。命令：
```bash
python -c "import json; d=json.load(open('Eldoria_V4.7.0.json', encoding='utf-8')); print(d.get('_meta',{}).get('version','unknown'))"
```

### Q3: 版本号递增错误（例如小修复却递增了次版本）怎么办？
**A**: 在 `build_eldoria.py` 中修正 `VERSION` 值，重新运行构建。旧的错误版本快照文件可以手动删除。

### Q4: 两个作者同时编辑产生冲突怎么办？
**A**: 每人在编辑前确认当前最新版本号，编辑时使用不同的工作副本。合并时根据「谁做了更大的变更」决定最终版本号，另一人的变更计入修订号。

### Q5: 备份文件太多，占用空间？
**A**: `backup/` 目录下的 `Eldoria_{timestamp}.json` 是时间戳备份。**建议保留最近 20 个**。更早的备份可以手动删除。注意：带版本号的发布快照（`Eldoria_V{VERSION}.json`）建议永久保留每个版本一份。
